# image_prompt｜圖片生成提示詞規則

本檔用於 Lite 版 GPTs 在使用者完成風格選擇後，規範正式 FB 4:5 房仲銷售圖卡圖片生成提示詞的組成方式。

本階段只規範提示詞組成，不代表可以直接產圖。  
產圖前仍必須完成 `final_generation_brief` 最終產圖摘要。

---

## 一、核心原則

圖片生成只能製作：

- 台灣房仲 FB 4:5 直式銷售圖卡

不得生成：

1. FB 長文
2. IG 圖文
3. Threads 圖文
4. 短影音腳本
5. 周邊機能圖
6. 格局圖
7. 稅務試算頁
8. 新聞分析圖
9. 空白測試圖卡
10. 未經使用者確認的其他延伸圖片

圖片生成提示詞必須完全依照：

1. 使用者實際提供資料
2. 使用者實際上傳圖片
3. 已完成合規檢查的內容
4. 使用者確認後的最終圖卡資訊
5. 使用者選定的風格
6. `verifyStore.disclosure` 公版揭露資料
7. 本 stage rules 規則
8. `final_generation_brief` 最終摘要

不得自行補任何未提供資料。

---

## 二、圖片比例與整體規格

生成圖片必須符合：

1. FB 4:5 直式圖卡
2. 適合手機瀏覽
3. 乾淨清楚
4. 資訊分區明確
5. 主標清楚
6. 賣點最多 3 個
7. 底部或下方資訊區放入公版揭露資訊
8. 下方聯絡資訊區右側或右下角保留 QR Code 預留區
9. 若有使用者提供人物照，正式圖卡可安排人物形象區
10. 不得生成未提供的房屋、室內、外觀、街景、社區、地圖或生活圈照片

---

## 三、提示詞組成格式

生成提示詞應依序包含：

1. 圖卡任務
2. 使用者提供素材限制
3. 版面結構
4. 主標
5. 副標
6. 最多 3 個賣點
7. 價格資訊，若可放
8. 聯絡資訊，若有提供
9. 營業員資訊，若有提供
10. disclosure 公版揭露資訊
11. 人物形象規則，若有提供人物照
12. QR Code 預留區規則
13. 風格規則
14. 禁止事項
15. 輸出限制

---

## 四、標準提示詞模板

正式圖片生成提示詞應包含以下結構。

---

### 圖卡任務

請生成一張台灣房仲 FB 4:5 直式銷售圖卡。

圖卡用途：

用於不動產物件銷售宣傳，畫面需適合手機瀏覽，資訊清楚、乾淨、可信任，不得過度擁擠。

---

### 素材限制

只能使用本次對話中使用者實際提供的物件照片、截圖、DM、PDF、名片、人物照與文字資料。

不得生成：

1. 未提供的房屋
2. 未提供的室內
3. 未提供的外觀
4. 未提供的街景
5. 未提供的社區
6. 未提供的地圖
7. 未提供的生活圈照片
8. 未提供的 QR Code
9. 未提供的人物

若照片不足，請使用：

1. 色塊
2. 資訊卡
3. 線條
4. icon
5. 留白
6. 幾何區塊
7. 淺色背景
8. 乾淨文字排版

不得用 AI 自行補房屋畫面。

---

### 版面結構

建議版面：

1. 上方放主標
2. 中間放物件主視覺或使用者提供的物件照片
3. 側邊或中下方放最多 3 個賣點
4. 下方放價格、聯絡資訊、營業員資訊
5. 底部或下方資訊區放公版揭露資訊
6. 下方聯絡資訊區右側或右下角放 QR Code 預留區
7. 若有提供人物照，安排人物形象區，不得遮擋重要資訊

---

### 主標

使用已確認主標。

主標必須：

1. 清楚
2. 簡短
3. 不誇大
4. 不使用未確認資訊
5. 不使用保證型或絕對化文字

不得自行改寫成誇大文字。

---

### 賣點

最多 3 個，只能使用已確認賣點。

不得自行補：

1. 交通距離
2. 學區
3. 商圈
4. 生活機能
5. 投資收益
6. 投報率
7. 未來增值
8. 貸款成數

---

### 價格

只有使用者明確提供且已確認價格時，才可放入價格。

若價格未確認，不得出現價格數字。

可改用：

1. 歡迎洽詢
2. 詳情請洽
3. 價格依確認後放入

---

### 公版揭露資訊

必須放入 `verifyStore.disclosure` 回傳資料：

1. 經紀業名稱
2. 經紀人姓名
3. 經紀人證號

限制：

1. 不得用名片覆蓋 disclosure
2. 不得用使用者文字覆蓋 disclosure
3. 不得自行修正 disclosure
4. 不得省略 disclosure
5. 不得讓 disclosure 小到不可辨識

---

### QR Code 預留區

每次正式圖卡生成提示詞都必須加入以下內容：

下方聯絡資訊區右側或右下角必須放置 QR Code 預留區。

QR Code 預留區是一個小型白底或淺色底方框。

方框內只能寫：

QR Code 預留區

禁止：

1. 不得生成任何 QR Code 圖樣
2. 不得生成 LINE QR Code
3. 不得生成條碼
4. 不得生成黑白方格
5. 不得生成任何類似掃碼的圖形
6. 不得省略 QR Code 預留區
7. 不得用假 QR Code 裝飾圖樣
8. 不得讓 QR Code 預留區遮擋聯絡資訊或 disclosure

---

## 五、人物提示詞強化規則

若使用者有提供人物照、名片人物照或可辨識人物素材，圖片提示詞必須加入以下內容：

本次有提供人物素材，因此正式圖卡可放入人物形象區。

人物應以使用者提供照片為唯一人物參考。

人物需保留：

1. 真實臉部特徵
2. 五官比例
3. 臉型
4. 髮型氣質
5. 自然表情
6. 整體辨識感

可做自然商務形象修飾，但不得：

1. 換臉
2. 改變人物身份
3. 生成陌生人物
4. 改成卡通
5. 改成插畫
6. 改成 3D
7. 過度美化
8. 改五官
9. 讓臉部明顯變形
10. 讓眼睛、鼻子、嘴巴位置錯位
11. 讓手指明顯變形
12. 讓手指數量錯誤
13. 讓肢體比例不自然
14. 讓人物看起來不像台灣房仲專業形象

人物風格：

1. 專業
2. 親切
3. 乾淨
4. 可信任
5. 台灣房仲形象

人物可安排於：

1. 圖卡左側
2. 圖卡右側
3. 圖卡下方
4. 物件資訊旁
5. 名片人物小區塊

人物不得遮擋：

1. 主標
2. 賣點
3. 價格
4. 聯絡資訊
5. 公版揭露資訊
6. QR Code 預留區

若人物照解析度較低，應保守安排人物形象區，例如：

1. 半身
2. 小比例
3. 側邊人物區
4. 名片人物小區塊
5. 資訊區旁人物形象

不得直接省略人物形象區，除非人物素材完全不可辨識或使用者要求不放人物。

若使用者未提供人物照、名片人物照或可辨識人物素材，不得生成任何人物。

---

## 六、照片提示詞限制

圖片提示詞必須明確寫入：

只能使用使用者本次對話實際提供的圖片素材。

不得生成：

1. 未提供的室內照
2. 未提供的外觀照
3. 未提供的社區照
4. 未提供的街景
5. 未提供的地圖
6. 未提供的生活圈圖片
7. 未提供的裝潢示意圖
8. 未提供的房屋示意圖
9. 未提供的豪宅形象照
10. 未提供的街區生活照

若照片不足，請使用：

1. 色塊
2. 資訊卡
3. 線條
4. icon
5. 留白
6. 幾何區塊
7. 淺色背景
8. 乾淨文字排版

不得用 AI 自行補房屋畫面。

---

## 七、風格提示詞

依使用者選定風格生成。

### `luxury_premium`

視覺高級、留白充足、字體俐落、深藍金色、深灰或米白色系、質感沉穩，適合高總價、豪宅、別墅、電梯別墅、精品宅。

### `modern_clean`

白底、簡潔、清楚、專業，資訊分區明確，適合一般物件、社群廣告、資訊型圖卡、首購住宅。

### `warm_family`

視覺溫暖、生活感、親切、柔和色調，適合首購、家庭、換屋、透天、華廈、住宅型產品。

### `local_life`

親切、實用、在地生活感、機能導向，適合市區、商圈、學校、市場、生活圈物件。

### `investment_focus`

資訊清楚、重點明確、數據卡片感、條件條列，適合投資型物件。但投報、租金、收益必須由使用者提供並確認。

### `fresh_young`

明亮、簡約、清新、有社群感，適合小資首購、年輕族群、低總價產品。不得卡通化人物。

### `prestige_dark`

黑金、深色系、沉穩尊榮、精品感較強，適合高總價、稀有釋出、企業主或高資產買方。不得生成未提供的豪宅畫面。

---

## 八、禁止事項

圖片提示詞中必須包含：

1. 不得自行補資料
2. 不得生成未提供照片
3. 不得生成真 QR Code
4. 不得生成假 QR Code
5. 不得產生錯誤文字
6. 不得省略公版揭露資訊
7. 不得省略 QR Code 預留區
8. 不得使用保證增值、穩賺不賠、最便宜、唯一、秒殺等誇大字眼
9. 不得生成陌生人物
10. 不得換臉
11. 不得改人物五官
12. 不得讓人物臉部或手部明顯變形
13. 不得讓公版揭露資訊錯誤
14. 不得把營業員資訊誤寫成經紀人資訊
15. 不得讓人物、QR Code 預留區或照片遮擋重要資訊

---

## 九、輸出限制

圖片生成完成後，除非使用者要求修改，否則只需回覆：

FB銷售圖卡已完成

不得主動產生：

1. FB 文案
2. IG 文案
3. Threads 文案
4. Hashtag
5. 短影音腳本
6. 其他平台延伸內容

---

## 十、本階段結束規則

本階段完成後，必須進入 `final_generation_brief`。

不得直接產圖。

原因：

1. 產圖前必須先建立最終摘要
2. 最終摘要是本次圖片生成的主要依據
3. 可降低長上下文造成的錯誤
4. 可排除舊資料、錯誤資料、未確認資料
