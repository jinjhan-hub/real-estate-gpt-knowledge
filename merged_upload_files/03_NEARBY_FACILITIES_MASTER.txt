# NEARBY FACILITIES MODULE｜周邊機能模組

本模組處理：

- 地址與座標確認
- 周邊生活機能分類
- API 查詢結果整理
- 周邊機能清單
- 周邊機能資訊卡文案

## 核心規則

- 台灣門牌地址不一定能被 Overpass 正確解析。
- 最穩定方式：使用者提供地址與 Google Maps 複製的經緯度。
- 若只有地址，需先請使用者補上 Google Maps 經緯度；若系統具備可用地理工具，才可協助轉換座標。
- 座標轉換後，必須請使用者確認。
- 地址與座標未確認前，不進入正式周邊機能查詢或正式輸出。
- 查詢結果應以「清單」或「資訊卡文案」呈現，不必主動生成街道示意圖。
- 若使用者沒有提供街道示意圖、地圖素材或手繪圖，不要主動生成街道示意圖。
- API 查詢不到的設施，不得自行補上。
- 不得自行創造學校、市場、公園、交通站點或商圈名稱。

## 標準輸入

- address：地址
- latitude：緯度
- longitude：經度
- coordinate_source：座標來源，例如使用者提供 / Google Maps / API 轉換
- coordinate_confirmation_status：未確認 / 已確認
- radius：查詢半徑，預設 1000 公尺
- categories：學校、交通、採買、公園、醫療、餐飲、便利商店、金融

## 標準輸出

- 地址
- 座標
- 座標來源
- 查詢半徑
- 分類清單
- 約略距離
- 生活圈觀察
- 可用於圖卡的短標語

## 距離呈現規則

- 距離需使用「約」。
- 若是座標計算，應寫成「依座標粗估約 X 公尺」。
- 不得把直線距離寫成實際路線距離。
- 沒有導航路線資料時，不得寫成精準步行時間或車程時間。
- 若要寫步行或車程，應使用「約略估算」語氣。

## 輸出限制

- 不得寫「走路就到」，除非有明確路線資料或使用者提供依據。
- 不得寫「生活機能最強」「明星學區」「蛋黃區」等未查證字眼。
- 不得把「附近有學校」延伸成「明星學區」。
- 不得把 API 回傳資料外的設施當成查詢結果。
- 若查詢結果不足，需如實說明「查詢結果有限」，不可自行補齊。

## 建議確認句

「我目前有地址與座標後，才能穩定查詢周邊機能。請確認這組座標是否正確，確認後我再整理生活機能清單或資訊卡文案。」

# COORDINATE POLICY｜地址與經緯度規則

## 優先輸入格式

周邊機能查詢最穩定的輸入方式，是請使用者同時提供：

1. 地址
2. Google Maps 複製的座標

範例：

地址：彰化縣埔心鄉中山路143號  
座標：23.940380, 120.559760

## 為什麼需要座標

- Overpass 對台灣門牌解析不穩定。
- 以座標查詢周邊 POI 較準。
- 可避免地址解析到錯誤地點。
- 地址可作為顯示與確認用途，座標作為查詢依據。

## 若使用者只提供地址

流程：

1. 先請使用者到 Google Maps 輸入地址。
2. 請使用者複製 Google Maps 顯示的經緯度。
3. 若系統具備可用地理工具，才可協助轉換座標。
4. 若由系統轉換座標，必須請使用者確認位置是否正確。
5. 地址與座標未確認前，不進入正式周邊機能查詢。

## 若使用者只提供座標

流程：

1. 檢查座標格式是否合理。
2. 詢問使用者是否要補上地址作為顯示名稱。
3. 若使用者不補地址，可使用「指定座標周邊」作為查詢名稱。
4. 查詢結果需標示為依座標整理。

## 座標格式規則

接受格式：

- `23.940380, 120.559760`
- `23.940380 120.559760`
- `緯度 23.940380，經度 120.559760`

## 台灣座標合理範圍

- 台灣緯度約 21 至 26
- 台灣經度約 119 至 123

若座標不在合理範圍內，需提醒使用者確認。

## 經緯度順序判斷

Google Maps 常見格式為：

緯度, 經度

例如：

23.940380, 120.559760

判斷方式：

- 第一個數字若落在 21 至 26，通常是緯度。
- 第二個數字若落在 119 至 123，通常是經度。
- 若第一個數字落在 119 至 123、第二個數字落在 21 至 26，可能是經緯度順序顛倒，需提醒使用者確認。

## 確認句

取得地址與座標後，需請使用者確認：

「我目前收到的地址是：＿＿＿＿  
座標是：＿＿＿＿  
請確認位置是否正確。確認後我再依這組座標整理周邊機能。」

## 禁止事項

- 不得只憑地址直接產出正式周邊機能。
- 不得在座標未確認前生成正式周邊機能圖或資訊卡。
- 不得自行推測地址所在位置。
- 不得把錯誤或未確認座標當成查詢依據。
- 不得將座標查詢結果寫成絕對精準距離。

# FACILITY CATEGORY RULES｜周邊機能分類規則

## 分類原則

周邊機能應依使用者提供座標、API 查詢結果或使用者確認資料整理。
不得自行創造設施名稱、距離、學區、商圈或交通節點。

若 API 查詢結果不足，需如實標示「查詢結果有限」，不得自行補齊。

---

## 固定分類

### 交通

可包含：

- 火車站
- 高鐵站
- 公車站
- 快速道路
- 交流道
- 主要道路
- 停車場

注意：

- 不得把「近主要道路」寫成「交通最便利」。
- 不得把直線距離寫成實際路線距離。
- 沒有導航資料時，不得寫成精準車程或步行時間。

---

### 採買

可包含：

- 超市
- 市場
- 便利商店
- 賣場
- 商圈
- 五金百貨
- 傳統商家

注意：

- 不得把「附近有市場」寫成「生活機能最強」。
- 不得把「商家較多」寫成「成熟商圈」，除非有資料依據。

---

### 教育

可包含：

- 幼兒園
- 國小
- 國中
- 高中
- 大學
- 補習班

注意：

- 不得把「附近有學校」寫成「明星學區」。
- 不得暗示一定可就讀特定學校。
- 學區與入學資格需以主管機關公告為準。

---

### 休閒

可包含：

- 公園
- 運動中心
- 圖書館
- 文化中心
- 活動中心
- 河堤或步道

注意：

- 不得把「附近有公園」寫成「綠地環繞」，除非實際資料明確。
- 不得誇大成「最適合親子」，除非有其他條件支撐。

---

### 醫療

可包含：

- 診所
- 藥局
- 醫院
- 牙醫
- 中醫
- 復健診所

注意：

- 不得把「附近有診所」寫成「醫療資源完善」，除非資料足夠。
- 醫療資訊僅作生活機能整理，不作醫療建議。

---

### 餐飲

可包含：

- 餐廳
- 咖啡廳
- 早餐店
- 小吃店
- 飲料店

注意：

- 餐飲類可列為生活便利參考，但不要過度放大成主要銷售賣點，除非物件本身是店面或店住。

---

### 金融

可包含：

- 銀行
- 郵局
- ATM
- 農會
- 信用合作社

注意：

- 不得把「附近有銀行」寫成「金融機能完整」，除非資料充足。
- 金融機能通常作為輔助資訊，不宜放成主賣點。

---

## 呈現規則

- 每類列 3 至 6 個即可。
- 若某類查詢結果不足，可以少列或標示「查詢結果有限」。
- 過多 POI 不要全部塞進文案。
- 優先列出與日常生活最相關的點。
- 距離使用「約」字。
- 若距離由座標計算，建議寫成「依座標粗估約 X 公尺」。
- 不要把車程、步行時間講得太絕對。
- 不要將未確認設施寫成確定存在。
- 不要將 API 查詢結果以外的內容補成查詢結果。

---

## 房仲文案轉換

查詢結果不要只列清單，也可以轉成生活語言，但語氣要保守。

可使用：

- 日常採買機能相對好安排
- 通勤動線可再依實際路線評估
- 基本生活需求在周邊可初步整理
- 適合重視生活便利性的買方進一步現場確認
- 周邊有學校、公園、採買點，可作為生活圈參考

避免使用：

- 採買完全不用繞路
- 交通超級方便
- 明星學區宅
- 生活機能最強
- 走路就到全部機能
- 親子首選
- 投資必看地段
