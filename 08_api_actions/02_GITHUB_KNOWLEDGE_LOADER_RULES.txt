# GITHUB KNOWLEDGE LOADER RULES｜GitHub Knowledge 載入規則

## 目的

用 GitHub 管理多個 GPT 共用的 Knowledge 檔案，避免每個 GPT 都塞大量內容。

本規則的目標是：

- 讓 GPT 依任務精準讀取所需檔案
- 避免一次載入全部 Knowledge
- 降低上下文負擔
- 維持輸出品質穩定
- 讓不同 GPT 可共用同一套資料庫

---

## 基本載入流程

1. 讀取 `manifest.json`
2. 讀取 `00_index/00_PROMPT_INDEX.txt`
3. 讀取 `00_index/01_ROUTING_TABLE.txt`
4. 依使用者任務判斷應載入的模組
5. 載入全域規則：
   - `01_global/00_GLOBAL_HARD_RULES.txt`
   - `01_global/01_CONFIRMATION_FLOW.txt`
   - `01_global/02_FACT_AND_SOURCE_POLICY.txt`
   - `01_global/03_OUTPUT_STYLE_BASELINE.txt`
6. 載入任務對應模組
7. 整理已知資料與缺漏資料
8. 請使用者確認
9. 執行任務

---

## 不要一次載入全部

除非使用者要求完整檢查、重建、更新或審核知識庫，不然不要一次讀取所有檔案。

應依任務載入：

- 銷售圖卡 / 文案 → `02_sales_generator/`
- 周邊機能 → `03_nearby_facilities/`
- 手繪格局圖 → `04_floorplan_converter/`
- 房市觀點文 → `06_market_content/`
- 稅務試算器 → `07_tax_calculator/`
- API / OpenAPI → `08_api_actions/`

---

## 主模組與輔助模組

若任務只屬於單一模組，只載入該模組與全域規則。

若任務跨模組，需先判斷主模組，再補讀輔助模組。

範例：

### 物件銷售圖卡 + 周邊機能

主模組：

- `02_sales_generator/`

輔助模組：

- `03_nearby_facilities/`

### 周邊機能 API 串接

主模組：

- `08_api_actions/`

輔助模組：

- `03_nearby_facilities/`

### 房市政策 FB 文案

主模組：

- `06_market_content/`

輔助模組：

- `01_global/02_FACT_AND_SOURCE_POLICY.txt`

### 稅務試算器開發

主模組：

- `07_tax_calculator/`

輔助模組：

- `08_api_actions/`，僅在需要 API 或 GitHub Pages 部署時載入

---

## 找不到對應模組時

若使用者任務無法明確對應模組：

1. 先讀取全域規則
2. 判斷是否需要使用者補充任務目標
3. 不要自行創造新模組
4. 不要硬套錯誤模組
5. 若任務不在本資料庫範圍內，需明確告知目前 Knowledge 未涵蓋

---

## 檔案命名規則

- 數字開頭，方便排序。
- 英文檔名，避免 URL 編碼問題。
- 內容使用繁體中文。
- 一個檔案只處理一類任務。
- 檔案名稱應清楚表示用途。
- 不要把多個任務混在同一個檔案。

---

## 檔案更新規則

更新 Knowledge 時：

- 不要直接刪除全域硬限制。
- 不要把案例寫進硬規則。
- 不要把一次性物件資料寫進通用規則。
- 不要把個人名片資訊寫成所有人通用預設值。
- 若修改 API 契約，需同步檢查 OpenAPI、Vercel API 與 GPT Actions。
- 若修改輸出模板，需確認不影響全域硬限制。

---

## 隨機性與一致性

Knowledge 不是為了讓每個輸出完全一樣。

它的作用是固定：

- 品質底線
- 任務流程
- 語氣方向
- 禁止事項
- 資料確認流程
- 平台差異
- 圖片生成限制

文案本身仍可依以下條件保留變化：

- 平台
- 物件類型
- 使用者指定風格
- 買方族群
- 物件特色
- 使用者語氣偏好

---

## 輸出原則

讀取 Knowledge 後，不要把 Knowledge 原文整段貼給使用者，除非使用者要求檢查、修改或複製檔案內容。

正常任務中，應將 Knowledge 轉換成：

- 可執行流程
- 清楚問題
- 整理後資料
- 文案
- 圖卡規格
- API 規格
- 開發建議
