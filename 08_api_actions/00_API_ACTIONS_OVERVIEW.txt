# API ACTIONS OVERVIEW｜API / OpenAPI / Vercel Actions 規則

本模組處理：

- GPT Actions
- OpenAPI Schema
- Vercel API
- Overpass 周邊機能查詢
- GitHub Knowledge 外部讀取架構
- API 錯誤處理與資料回傳規則

---

## 核心原則

- API 只做資料查詢、資料整理、格式轉換或特定運算。
- API 不負責生成使用者未提供的圖片。
- API 查不到資料時，應回報查不到，不得自行補資料。
- API 回傳外的內容，不得被 GPT 當成查詢結果。
- GPT Instructions 應寫清楚什麼時候調 API，什麼時候讀 Knowledge。
- OpenAPI Schema 需保持簡潔、可驗證、可被 GPT Actions 正確讀取。
- 修改 API 或 OpenAPI 前，需確認是否會影響既有 GPTs Action。

---

## GPT 與 Knowledge 分工

### Instructions

放：

- 全域硬限制
- 任務流程
- 工具調用順序
- 不可違反規則
- 什麼情況要先確認使用者資料
- 什麼情況要呼叫 API
- 什麼情況只讀 Knowledge

### Knowledge

放：

- 模板
- 風格庫
- 案例
- 細節規則
- 可替換資料
- 輸出格式
- 禁止事項
- 模組化流程

### API

放：

- 周邊機能查詢
- 座標查詢
- 外部資料讀取
- 特定運算
- 結構化資料回傳
- 錯誤訊息
- 查詢失敗狀態

---

## API 使用限制

- API 回傳資料需如實呈現。
- 不得把 API 沒回傳的設施、距離、地址、學校、商圈自行補上。
- 不得把 API 估算距離寫成絕對精準距離。
- 不得把 Overpass 查詢結果包裝成 Google Maps 導航結果。
- 若 API 回傳資料不足，應標示「查詢結果有限」。
- 若 API 發生錯誤，需回覆錯誤原因或請使用者補資料，不得假裝查詢成功。

---

## OpenAPI 維護規則

修改 OpenAPI Schema 時：

- 不得任意更改 endpoint。
- 不得任意更改 operationId。
- 不得任意刪除 requestBody 欄位。
- 不得任意刪除 response 欄位。
- 不得任意更改 required 欄位。
- 不得破壞 JSON 格式。
- 不得讓 schema 無法通過 GPT Actions 驗證。
- 若新增欄位，需保持向下相容。
- 若更改欄位名稱，需同步修改 API 程式與 GPT Instructions。

---

## Vercel API 維護規則

- Vercel API 應保持 endpoint 穩定。
- 環境變數不可寫死在前端。
- API 錯誤需回傳清楚訊息。
- 若外部服務失敗，需回傳可理解的錯誤狀態。
- 不得讓 API 回傳空白結果卻顯示成功。
- 若 Overpass 查詢逾時，需回傳 timeout 或查詢失敗訊息。
- 若座標格式錯誤，需回傳座標錯誤訊息。

---

## 避免影響既有 API

只要以下項目沒有被改掉，新增 Knowledge 結構通常不會影響原本串接：

- API endpoint
- operationId
- request schema
- response schema
- required 欄位
- GPT Actions 設定
- Vercel 部署網址

若改 Instructions 中的工具調用邏輯，才可能影響 API 使用方式。

## 錯誤處理原則

常見錯誤需明確處理：

### 1. 缺少地址

回覆：

「目前缺少地址，無法建立顯示名稱。請提供物件地址。」

### 2. 缺少座標

回覆：

「目前缺少經緯度，無法穩定查詢周邊機能。請提供 Google Maps 複製的座標。」

### 3. 座標格式錯誤

回覆：

「座標格式可能有誤，請確認是否為緯度、經度，例如 23.940380, 120.559760。」

### 4. API 查無資料

回覆：

「目前 API 查詢結果有限，無法確認足夠周邊設施，不會另外補上未查證資料。」

### 5. 外部服務逾時

回覆：

「外部查詢服務暫時無回應，建議稍後再查，或先改用使用者提供資料整理。」

---

## 安全輸出原則

API 結果輸出給使用者時，需整理成可讀格式，例如：

- 分類清單
- 約略距離
- 生活圈觀察
- 資訊卡文案

不要直接把原始 JSON 丟給一般使用者，除非使用者明確要求看 API 回傳資料。
