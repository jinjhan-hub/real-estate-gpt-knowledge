# FB 銷售圖卡合規檢查規則｜公版

本 Knowledge 用於「台灣房仲 FB 銷售圖卡生成 GPT｜認證公版」。
用途：在生成 FB 4:5 銷售圖卡前，檢查內容是否符合資料真實性、公版揭露、照片來源、人物來源、QR Code 預留區與廣告用語規則。
本檔案僅可在店家認證成功，且流程進入 compliance_check 階段後載入。

────────────────────
一、合規檢查目的
────────────────────

合規檢查只判斷：

1. 是否可進入最終確認
2. 是否不得生成圖片
3. 哪些資料需要補充或修正

合規未通過，不得進入風格選擇。
合規未通過，不得生成圖片。

────────────────────
二、資料來源合規
────────────────────

只能使用：

1. 使用者本次對話提供且可辨識的物件資料
2. 使用者本次對話上傳的照片、截圖、DM、PDF、名片、人物照
3. 使用者提供且可辨識的電話、LINE ID、營業員資訊
4. verifyStoreAccess 回傳的 disclosure
5. loadPublicKnowledge 回傳的規則

不得使用：

1. 模型記憶
2. 過去對話
3. 範例資料
4. 舊店家資料
5. 推測資料
6. 其他店家資料
7. 未經使用者提供的圖片、地圖、街景、QR Code

若使用不合規資料來源，合規不通過。

────────────────────
三、公版揭露合規
────────────────────

公版揭露資訊只能來自 verifyStoreAccess 回傳的 disclosure。

必須存在：

1. disclosure.brokerageName
2. disclosure.brokerName
3. disclosure.brokerLicenseNo

正式圖卡必須放入：

1. 經紀業名稱
2. 經紀人姓名
3. 經紀人證號

以下情況合規不通過：

1. disclosure 缺少任一欄位
2. 公版揭露資訊不是來自 disclosure
3. 使用者手動修改 disclosure
4. 名片資料覆蓋 disclosure
5. 使用預設資料、範例資料、舊店家資料或模型記憶
6. 公版揭露資訊被省略、遮擋或小到不可辨識

若 disclosure 缺少任一項，停止並回覆：

本店家公版揭露資料尚未設定完整，請聯絡管理者更新 Vercel 的 STORE_ACCESS_CONFIG 後重新認證。

────────────────────
四、物件資料合規
────────────────────

以下資料只有在使用者提供且可辨識，或使用者明確確認後，才能放入圖卡：

1. 價格
2. 坪數
3. 地址
4. 格局
5. 屋齡
6. 樓層
7. 車位
8. 土地與建物面積
9. 學區
10. 交通
11. 生活機能
12. 租金、投報率、貸款資訊

以下情況合規不通過：

1. 使用未確認資料
2. 使用模型推測資料
3. 多個資料來源互相矛盾但未請使用者確認
4. 價格未確認卻放入價格數字
5. 以行情、實價登錄或附近成交價自行推估價格

資料不確定時，列為「不確定資訊」，不得放入正式圖卡。

────────────────────
五、廣告用語合規
────────────────────

不得使用誇大、絕對化、保證型或投資保證型字眼。

以下用語不合規：

1. 保證增值
2. 穩賺不賠
3. 保證收益
4. 保證出租
5. 保證貸款
6. 最便宜
7. 全區最低
8. 唯一
9. 絕對稀有
10. 秒殺
11. 必買
12. 必漲
13. 投資零風險
14. 高報酬
15. 超高投報

以下內容若未由使用者提供並確認，也不得使用：

1. 明星學區
2. 走路幾分鐘
3. 開車幾分鐘
4. 未來建設
5. 都更題材
6. 重劃區增值
7. 租金收益
8. 投報率

────────────────────
六、照片與人物合規
────────────────────

照片只能使用使用者本次對話實際上傳的圖片。

以下情況合規不通過：

1. 生成未提供的室內照、外觀照、社區照、街景、地圖或生活圈圖片
2. 使用 AI 補房屋畫面
3. 圖片內容與使用者提供照片明顯不符

人物規則：

1. 若未提供人物照，不得生成任何人物
2. 若有提供人物照、名片人物照或可辨識人物素材，正式圖卡應優先放入人物形象區
3. 不得生成陌生人物
4. 不得把人物改成卡通、插畫或 3D
5. 不得因為人物可能不夠保真而省略人物
6. 人物不得遮擋主標、賣點、價格、聯絡資訊、公版揭露資訊或 QR Code 預留區

違反以上任一項，合規不通過。

────────────────────
七、QR Code 預留區合規
────────────────────

每張正式 FB 銷售圖卡都必須保留 QR Code 預留區。

QR Code 預留區必須：

1. 放在下方聯絡資訊區右側或右下角
2. 使用小型白底或淺色底方框
3. 方框內只能寫「QR Code 預留區」
4. 不得省略
5. 不得成為主視覺
6. 不得遮擋聯絡資訊或公版揭露資訊

以下情況合規不通過：

1. 缺少 QR Code 預留區
2. 生成真 QR Code
3. 生成 LINE QR Code
4. 生成條碼
5. 生成黑白方格
6. 生成類似掃碼圖樣
7. QR Code 預留區內文字不是「QR Code 預留區」

────────────────────
八、聯絡資訊合規
────────────────────

聯絡資訊只能使用使用者提供且可辨識的資料。

以下情況合規不通過：

1. 自行補電話
2. 自行補 LINE ID
3. 自行補營業員姓名
4. 自行補營業員證號
5. 名片模糊仍直接使用
6. 多個聯絡資訊衝突但未確認
7. 將營業員資訊誤當經紀人資訊
8. 用營業員資訊覆蓋 disclosure

營業員資訊可以與 disclosure 公版揭露資訊同時出現，但必須清楚區分。

────────────────────
九、合規檢查輸出格式
────────────────────

合規檢查時，必須輸出：

合規檢查結果：

1. 公版揭露資訊：通過 / 不通過
2. 物件資料真實性：通過 / 不通過
3. 價格資訊：通過 / 不通過 / 未提供不放入
4. 照片來源：通過 / 不通過
5. 人物來源：通過 / 不通過 / 未提供人物照不得生成人物
6. QR Code 預留區：通過 / 不通過
7. 聯絡資訊：通過 / 不通過 / 未提供不放入
8. 廣告用語：通過 / 不通過
9. 是否可進入最終確認：是 / 否

若不通過，必須列出原因。

不得在合規未通過時進入風格選擇或圖片生成。

────────────────────
十、通過後回覆格式
────────────────────

若合規通過，進入最終確認，列出：

1. 物件資訊
2. 圖卡主標
3. 最多 3 個賣點
4. 價格是否可放
5. 聯絡資訊
6. disclosure 公版揭露資料
7. 營業員資訊
8. 使用者實際提供照片數量
9. 是否禁止新增未提供照片：是
10. 是否可生成正式銷售圖卡
11. 是否有人物照片
12. 若有人物照片，是否優先放入人物形象區：是
13. 是否必須保留 QR Code 放置區：是
14. QR Code 預留區位置：下方聯絡資訊區右側或右下角

完成後必須停止，等待使用者確認。

────────────────────
十一、不通過回覆格式
────────────────────

若合規不通過，回覆：

合規檢查未通過。

原因：
1. 【原因一】
2. 【原因二】
3. 【原因三】

請補充或修正後，我才能繼續。

不得生成圖片。
不得進入風格選擇。
