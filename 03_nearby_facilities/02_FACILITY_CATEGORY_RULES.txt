# FACILITY CATEGORY RULES｜周邊機能分類規則

## 分類原則

周邊機能應依使用者提供座標、API 查詢結果或使用者確認資料整理。
不得自行創造設施名稱、距離、學區、商圈或交通節點。

若 API 查詢結果不足，需如實標示「查詢結果有限」，不得自行補齊。

---


## API 固定查詢類別

目前 API 固定支援的查詢類別只有：

- `school`：教育 / 學校
- `transport`：交通
- `shopping`：採買
- `park`：公園 / 休閒
- `medical`：醫療

每次 API 呼叫預設一次查詢五種固定 categories：school、transport、shopping、park、medical。
不得將餐飲、金融、便利商店等分類當成 API 固定支援類別，除非 API 已更新。
若使用者提供餐飲、金融或便利商店資訊，可放入補充整理，但必須標示其資料來源。

---

## 固定分類

### 交通

可包含：

- 火車站
- 高鐵站
- 公車站
- 快速道路
- 交流道
- 主要道路
- 停車場

注意：

- 不得把「近主要道路」寫成「交通最便利」。
- 不得把直線距離寫成實際路線距離。
- 沒有導航資料時，不得寫成精準車程或步行時間。

---

### 採買

可包含：

- 超市
- 市場
- 便利商店
- 賣場
- 商圈
- 五金百貨
- 傳統商家

注意：

- 不得把「附近有市場」寫成「生活機能最強」。
- 不得把「商家較多」寫成「成熟商圈」，除非有資料依據。

---

### 教育

可包含：

- 幼兒園
- 國小
- 國中
- 高中
- 大學
- 補習班

注意：

- 不得把「附近有學校」寫成「明星學區」。
- 不得暗示一定可就讀特定學校。
- 學區與入學資格需以主管機關公告為準。

---

### 休閒

可包含：

- 公園
- 運動中心
- 圖書館
- 文化中心
- 活動中心
- 河堤或步道

注意：

- 不得把「附近有公園」寫成「綠地環繞」，除非實際資料明確。
- 不得誇大成「最適合親子」，除非有其他條件支撐。

---

### 醫療

可包含：

- 診所
- 藥局
- 醫院
- 牙醫
- 中醫
- 復健診所

注意：

- 不得把「附近有診所」寫成「醫療資源完善」，除非資料足夠。
- 醫療資訊僅作生活機能整理，不作醫療建議。

---

### 餐飲

可包含：

- 餐廳
- 咖啡廳
- 早餐店
- 小吃店
- 飲料店

注意：

- 餐飲類可列為生活便利參考，但不要過度放大成主要銷售賣點，除非物件本身是店面或店住。

---

### 金融

可包含：

- 銀行
- 郵局
- ATM
- 農會
- 信用合作社

注意：

- 不得把「附近有銀行」寫成「金融機能完整」，除非資料充足。
- 金融機能通常作為輔助資訊，不宜放成主賣點。

---

## 呈現規則

- 每類列 3 至 6 個即可。
- 若某類查詢結果不足，可以少列或標示「查詢結果有限」。
- 過多 POI 不要全部塞進文案。
- 優先列出與日常生活最相關的點。
- 距離使用「約」字。
- 若距離由座標計算，建議寫成「依座標粗估約 X 公尺」。
- 不要把車程、步行時間講得太絕對。
- 不要將未確認設施寫成確定存在。
- 不要將 API 查詢結果以外的內容補成查詢結果。

---

## 房仲文案轉換

查詢結果不要只列清單，也可以轉成生活語言，但語氣要保守。

可使用：

- 日常採買機能相對好安排
- 通勤動線可再依實際路線評估
- 基本生活需求在周邊可初步整理
- 適合重視生活便利性的買方進一步現場確認
- 周邊有學校、公園、採買點，可作為生活圈參考

避免使用：

- 採買完全不用繞路
- 交通超級方便
- 明星學區宅
- 生活機能最強
- 走路就到全部機能
- 親子首選
- 投資必看地段
