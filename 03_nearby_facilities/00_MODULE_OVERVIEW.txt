# NEARBY FACILITIES MODULE｜周邊機能模組

本模組處理：

- 地址與座標確認
- 周邊生活機能分類
- API 查詢結果整理
- 周邊機能清單
- 周邊機能資訊卡文案

## 核心規則

- 台灣門牌地址不一定能被 Overpass 正確解析。
- 最穩定方式：使用者提供地址與 Google Maps 複製的經緯度。
- 若只有地址，需先請使用者補上 Google Maps 經緯度；若系統具備可用地理工具，才可協助轉換座標。
- 座標轉換後，必須請使用者確認。
- 地址與座標未確認前，不進入正式周邊機能查詢或正式輸出。
- 查詢結果應以「清單」或「資訊卡文案」呈現，不必主動生成街道示意圖。
- 若使用者沒有提供街道示意圖、地圖素材或手繪圖，不要主動生成街道示意圖。
- API 查詢不到的設施，不得自行補上。
- 不得自行創造學校、市場、公園、交通站點或商圈名稱。

## 標準輸入

- address：地址
- lat：緯度
- lng：經度
- coordinate_source：座標來源，例如使用者提供 / Google Maps / API 轉換
- coordinate_confirmation_status：未確認 / 已確認
- radius：查詢半徑，預設 500 公尺
- categories：school、transport、shopping、park、medical（預設一次查詢五種固定類別）


## API 類別限制

目前周邊機能 API 固定支援以下 categories：

- school：教育 / 學校
- transport：交通
- shopping：採買
- park：公園 / 休閒
- medical：醫療

每次呼叫 API 預設一次查詢五種固定 categories：school、transport、shopping、park、medical。
不得自行加入 food、finance 或其他未支援分類，除非 API 已更新並同步修改 OpenAPI 與本 Knowledge。

餐飲、金融、便利商店等內容，只有在使用者提供資料、API 實際回傳或另有已確認來源時，才可作為補充文案整理，不得當成 API 固定查詢類別。

## 標準輸出

- 地址
- 座標
- 座標來源
- 查詢半徑
- 分類清單
- 約略距離
- 生活圈觀察
- 可用於圖卡的短標語

## 距離呈現規則

- 距離需使用「約」。
- 若是座標計算，應寫成「依座標粗估約 X 公尺」。
- 不得把直線距離寫成實際路線距離。
- 沒有導航路線資料時，不得寫成精準步行時間或車程時間。
- 若要寫步行或車程，應使用「約略估算」語氣。

## 輸出限制

- 不得寫「走路就到」，除非有明確路線資料或使用者提供依據。
- 不得寫「生活機能最強」「明星學區」「蛋黃區」等未查證字眼。
- 不得把「附近有學校」延伸成「明星學區」。
- 不得把 API 回傳資料外的設施當成查詢結果。
- 若查詢結果不足，需如實說明「查詢結果有限」，不可自行補齊。

## 建議確認句

「我目前有地址與座標後，才能穩定查詢周邊機能。請確認這組座標是否正確，確認後我再整理生活機能清單或資訊卡文案。」
